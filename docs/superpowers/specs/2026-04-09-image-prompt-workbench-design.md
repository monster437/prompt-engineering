# 图片提示词工作台设计文档

## 1. 项目目标
构建一个支持浏览器可视化操作的图片提示词工作台。用户输入一句自然语言后，系统通过接入兼容 OpenAI 风格接口的 AI 文本模型，帮助用户生成更高质量的图片提示词。

第一阶段以“提示词助手优先”为核心，不把直接出图作为主流程，但系统必须从架构上预留图片生成能力。应用需要支持 Docker Compose 部署。

## 2. 第一阶段范围
### 包含
- 浏览器可视化 Web 应用
- Next.js 单体应用架构
- AI 配置界面
  - 文本配置
  - 图像配置
- 每条配置支持：供应商名、Base URL、API Key、模型名
- 模型选择能力
  - 选择用于生成提示词的文本模型
  - 选择目标模型类型（通用 / 即梦 / Grok / Nano Banana 等）
  - 模型名后展示所属供应商
- 多工作台标签并发
- 两种提示词生成模式
  - 智能追问模式
  - 一键优化模式
- 输出语言切换：中文 / 英文
- 结果展示
  - 最终提示词
  - 参数摘要（风格、场景、时间、氛围、质量、构图等）
- 二次微调
- 后端代理兼容 OpenAI 风格文本接口
- 为图片生成与图片下载能力预留结构
- Docker Compose 部署

### 不包含
- 第一阶段不把直接图片生成作为主流程
- 不做视频生成
- 不做历史记录系统
- 不做多用户系统
- 不做权限系统

## 3. 用户核心场景
### 场景一：一键优化
用户输入一句简短描述，选择调用文本模型、目标模型类型、输出语言后，系统直接生成高质量提示词和参数摘要。

### 场景二：智能追问
用户输入一句简短描述，系统先判断信息是否充足；如果不充足，则动态追问最关键的信息，直到足够生成最终提示词。

### 场景三：结果微调
用户在已有结果基础上继续输入修改指令，例如“改成黄昏”“更有电影感”，系统基于当前上下文重新生成提示词和参数摘要。

### 场景四：多工作台并发
单个用户可同时打开多个工作台标签。每个工作台都可独立选择不同模型、不同目标类型，并同时发起提示词生成任务，互不影响。

## 4. 整体架构
系统采用 Next.js 单体应用架构，分为三层：

### 4.1 前端工作台层
负责提供浏览器可视化体验，包括：
- 多工作台标签管理
- 输入区
- 模式选择
- 模型选择
- AI 配置弹窗
- 追问交互
- 结果展示
- 微调操作

### 4.2 后端服务层
由 Next.js API 路由提供，负责：
- 保存和读取 AI 配置
- 返回可选模型列表
- 接收提示词生成请求
- 接收提示词微调请求
- 未来接收图片生成请求
- 安全保存 API Key
- 代理请求外部 AI 服务

### 4.3 模型适配与提示词编排层
负责：
- 根据用户所选工作模式、一键优化/智能追问、输出语言、目标模型类型组装系统 prompt
- 屏蔽不同兼容 OpenAI 供应商在 `/v1/chat/completions` 与 `/v1/responses` 上的差异
- 统一返回项目内部标准格式
- 为未来的 `/v1/images/generations` 与基于 `/v1/chat/completions` 的图片生成接口做统一适配

## 5. 前端信息架构
### 5.1 顶部工具栏
包含：
- AI 配置按钮
- 新建工作台按钮
- 删除当前工作台按钮
- 工作台数量或标签总览

### 5.2 工作台标签栏
采用标签页结构，每个标签对应一个独立会话，例如：
- 工作台 1
- 工作台 2
- 工作台 3

用户可快速切换工作台，不需要翻页，也不使用多栏布局作为第一阶段主形态。

### 5.3 当前工作台内容区
每个工作台独立维护：
- 原始输入
- 生成模式（智能追问 / 一键优化）
- 输出语言（中文 / 英文）
- 调用文本模型
- 目标模型类型
- 追问过程
- 最终提示词
- 参数摘要
- 微调输入
- 当前任务状态
- 未来图片能力相关状态占位

### 5.4 AI 配置弹窗
分为两个页签：
- 文本配置
- 图像配置

每条配置包含：
- 供应商名称
- Base URL
- API Key
- 模型名列表

模型选择界面展示格式示例：
- `gpt-4.1-mini (OpenRouter)`
- `deepseek-chat (供应商A)`

## 6. 核心概念澄清
系统中存在两类不同概念：

### 6.1 调用模型
指真正执行“生成提示词”工作的 AI 文本模型，由用户在已配置的文本模型中选择。

### 6.2 目标模型类型
指最终提示词准备投喂给哪一类图像/多模态模型使用，例如：
- 通用
- 即梦
- Grok
- Nano Banana

调用模型决定“谁来帮用户写”，目标模型类型决定“提示词写成给谁使用的风格”。两者需要同时可选。

## 7. 数据设计
### 7.1 AI 配置
每条配置建议包含：
- `id`
- `type`：`text` / `image`
- `providerName`
- `baseURL`
- `apiKey`
- `models[]`

模型对象建议包含：
- `modelName`
- `label`
- `providerId`

### 7.2 工作台状态
每个工作台建议维护：
- `workspaceId`
- `title`
- `mode`：`interview` / `optimize`
- `outputLanguage`：`zh` / `en`
- `selectedTextModel`
- `selectedTargetType`
- `selectedImageModel`（预留）
- `sourcePrompt`
- `questionMessages[]`
- `answers[]`
- `finalPrompt`
- `parameterSummary`
- `refineInstruction`
- `status`：`idle / generating / asking / refining / error`

### 7.3 并发任务状态
独立维护任务表，建议包含：
- `taskId`
- `workspaceId`
- `requestType`
- `status`
- `startedAt`
- `error`

目的：确保多个工作台同时运行时，状态互不覆盖。

## 8. 后端接口设计
### 8.1 配置管理接口
- `GET /api/configs`
- `POST /api/configs`
- `PUT /api/configs/:id`
- `DELETE /api/configs/:id`

### 8.2 模型聚合接口
- `GET /api/models`

用于返回模型与供应商归属关系，供前端选择。

### 8.3 提示词生成接口
- `POST /api/prompt/generate`

根据工作台模式决定：
- 直接输出最终提示词
- 或返回需继续追问的问题

### 8.4 提示词微调接口
- `POST /api/prompt/refine`

基于当前工作台已有上下文继续优化结果。

### 8.5 图片生成接口（预留）
- `POST /api/image/generate`

第一阶段仅预留设计，不作为主流程实现目标。

## 9. 提示词生成流程
### 9.1 一键优化模式
流程：
1. 用户输入自然语言描述
2. 用户选择调用文本模型
3. 用户选择目标模型类型
4. 用户选择输出语言
5. 系统直接生成最终提示词与参数摘要

默认采用强力优化策略，尽量生成专业、细节丰富、成片感更强的提示词。

### 9.2 智能追问模式
流程：
1. 用户输入自然语言描述
2. 模型判断信息是否足够
3. 若不足，则动态追问最关键问题
4. 用户回答后继续判断
5. 当信息足够时生成最终提示词与参数摘要

追问为智能动态生成，不采用固定字段表单。

### 9.3 二次微调模式
流程：
1. 读取当前工作台已有结果
2. 用户输入新的修改指令
3. 系统在原结果基础上重新生成最终提示词与参数摘要

## 10. 返回格式统一原则
无论是一键优化、智能追问还是二次微调，后端都应返回统一结构，至少包含：
- 当前状态：`needs_clarification` / `completed`
- AI 追问内容（如有）
- 最终提示词（如有）
- 参数摘要
- 当前工作台上下文快照

这样前端不需要为每种模式维护完全不同的渲染和状态管理逻辑。

## 11. 并发设计
### 11.1 并发粒度
并发以“工作台”为最小单位，而不是整个页面全局串行。

### 11.2 目标行为
允许：
- 工作台 1 正在生成提示词
- 工作台 2 同时使用不同文本模型生成另一份提示词
- 两者互不阻塞

### 11.3 第一阶段并发边界
第一阶段支持：
- 多工作台独立并发请求
- 每个工作台同一时刻只运行一个主任务

不要求第一阶段实现同一个工作台内的多任务并行。

## 12. 安全设计
### 12.1 安全目标
虽然 AI 配置入口在浏览器前端，但实际模型调用不能由前端直接携带 API Key 请求外部 AI 服务。

### 12.2 推荐方案
采用“前端配置 + 后端保存 + 后端代理请求”的结构：
- 前端负责输入和管理供应商配置
- 后端负责保存完整配置
- 后端负责实际调用外部模型
- 前端仅通过配置 ID 和模型选择发起业务请求

### 12.3 为什么不采用前端直连
前端直连会带来：
- API Key 暴露风险
- 跨域问题
- 多供应商兼容逻辑分散到前端
- 错误处理复杂
- 后续接图片模型难以统一适配

### 12.4 第一阶段最低安全边界
至少做到：
- 前端不直接请求外部模型
- API Key 不明文回显
- 服务端日志不打印 API Key
- 配置读取接口只返回脱敏 key
- 工作台请求通过配置 ID 发起，而不是反复传递完整 key

### 12.5 更稳妥的增强方案
可在服务端对 API Key 进行加密存储，并在运行时解密后调用外部模型。

## 13. 图片能力预留
虽然第一阶段不把图片生成为主流程，但必须预留：
- 图像配置管理
- 图像模型选择
- 图片生成接口结构
- 图片结果展示位
- 图片下载按钮位置

当第二阶段接入图片生成后，结果区至少支持：
- 单张图片展示
- 下载按钮

如果未来支持多图，再扩展为逐张下载。

## 14. Docker Compose 部署
第一阶段要求支持 Docker Compose 部署。

建议至少包含：
- `web`：Next.js 应用服务
- `db`：用于保存配置与必要业务数据的持久化存储

部署要求：
- 配置数据可持久化
- 重启容器后数据不丢失
- 可通过环境变量配置基础运行参数

## 15. 测试重点
第一阶段至少覆盖以下验证：

### 15.1 配置管理
- 文本配置和图像配置可新增、编辑、删除
- API Key 脱敏展示正常
- 模型与供应商映射正确

### 15.2 多工作台并发
- 工作台之间状态独立
- 不同工作台可同时发起请求
- 不同工作台可选不同模型

### 15.3 提示词流程
- 一键优化流程正常
- 智能追问流程正常
- 二次微调流程正常
- 输出语言切换正常
- 目标模型类型切换正常

### 15.4 图片能力预留
- 图像配置与图像模型结构可管理
- 图片结果区可扩展
- 图片下载入口位置已预留

## 16. 推荐实施结论
第一阶段应聚焦于“高质量提示词生成工作台”，把以下能力做扎实：
- 可视化配置
- 多工作台并发
- 提示词生成体验
- 安全的模型代理
- 可扩展的图像能力基础结构

这样第二阶段接入图片生成时，无需推翻第一阶段架构，只需在既有工作台体系上补全图片生成与下载流程。
